# "Task 기반의 상태 머신(State Machine)"
async/await는 단순히 비동기를 쉽게 쓰기 위한 문법 설탕(Syntactic Sugar)이 아니라, 컴파일러가 비동기 메서드를 하나의 '상태 머신'으로 변환해주는 메커니즘입니다.

- Async: 메서드 내부에서 await를 사용할 수 있음을 알리고, 메서드 전체를 비동기 처리를 위한 상태 머신 구조로 변경합니다.

- Await: 해당 비동기 작업(Task)이 완료될 때까지 현재 메서드의 실행을 일시 중단하고, 제어권을 호출자(Caller)에게 반환합니다. 작업이 완료되면 중단된 지점부터 다시 실행을 이어갑니다.


----

### async void는 왜 지양해야 하는가?

async void는 반환할 Task 객체가 없기 때문에 호출자가 작업의 완료 시점이나 예외 발생 여부를 추적할 수 없습니다.

 특히 예외가 발생하면 호출자가 아닌 SynchronizationContext의 최상단에서 처리되어 애플리케이션이 크래시(Crash)될 위험이 큽니다.
 
  이벤트 핸들러를 제외하고는 항상 async Task를 사용해야 합니다.

### ConfigureAwait(false)는 왜 쓰는가?
답변: 비동기 작업이 끝난 후 다시 원래의 스레드(주로 UI 스레드)로 복귀하지 않도록 합니다.

 라이브러리 개발 시 사용하면 교착 상태(Deadlock)를 방지하고 성능을 미세하게 향상시킬 수 있습니다.
 
  UI와 상관없는 백그라운드 연산이라면 false를 쓰는 것이 권장됩니다.

### 설명용
단순히 비동기를 쉽게 쓰는 도구로 알고 있습니다

내부적으로는 컴파일러가 메서드를 상태 머신으로 변환하여   
await 지점에서 제어권을 반환하고,
 작업 완료 후 저장된 상태를 복원하여 로직을 이어가는 구조를 이해하고 있습니다

 ```
 Syntactic Sugar: 컴파일러가 만들어내는 상태 머신 구조.

Non-blocking: UI 스레드 차단 방지.

Continuation: 작업 완료 후 실행할 나머지 코드.

Task/ValueTask: 비동기 작업의 결과 및 완료 상태를 나타내는 객체.
```
